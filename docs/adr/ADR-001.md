# ADR-001: Escolha do RabbitMQ como Broker de Mensageria

**Status:** Aceito  
**Data:** 2026-04-23  
**Autores:** Time de Arquitetura

---

## Contexto

O AntiFraude Engine precisa desacoplar a recepção de transações (API síncrona) do processamento das regras de fraude (assíncrono). As opções consideradas foram:

- **RabbitMQ** — broker de mensageria AMQP, maduro e amplamente adotado
- **Apache Kafka** — plataforma de streaming distribuído, alto throughput
- **Azure Service Bus** — serviço gerenciado de mensageria em nuvem
- **Amazon SQS** — fila gerenciada AWS

O sistema precisa rodar localmente via Docker Compose e ser preparado para Kubernetes, sem dependência obrigatória de provedores de nuvem na fase inicial.

---

## Decisão

**RabbitMQ** foi escolhido como broker de mensageria, com as seguintes configurações:

- Exchange do tipo `direct` para roteamento determinístico de transações
- Dead Letter Queue (DLQ) configurada via MassTransit para mensagens que falham após 3 tentativas
- MassTransit como camada de abstração (permite trocar o broker no futuro sem alterar consumers)
- Imagem `rabbitmq:3.13-management-alpine` com Management UI ativada

---

## Consequências

### Positivas

- **Simplicidade operacional:** fácil de rodar localmente com uma única imagem Docker; sem dependência de cloud
- **Routing flexível:** exchanges do tipo `direct`, `topic` e `fanout` cobrem todos os casos de uso previstos
- **Management UI:** visibilidade imediata de filas, mensagens pendentes e DLQ sem ferramentas adicionais
- **MassTransit abstraction:** a troca para Kafka ou Azure Service Bus exige apenas mudança de configuração, não de código de consumer
- **Maturidade:** protocolo AMQP 0-9-1 estável; ampla documentação e suporte da comunidade

### Negativas

- **Não é um log:** RabbitMQ não oferece replay de mensagens históricas (ao contrário do Kafka); transações já consumidas não podem ser relidas sem persistência adicional
- **Escalabilidade horizontal limitada:** Kafka supera RabbitMQ em throughput para volumes muito altos (>100k msg/s); para o volume atual, RabbitMQ é suficiente
- **Clustering:** configuração de cluster RabbitMQ é mais complexa do que um cluster Kafka gerenciado

### Neutras

- A DLQ deve ser monitorada ativamente; mensagens na DLQ indicam falhas que requerem atenção manual
- Em produção, recomenda-se RabbitMQ em modo cluster (mínimo 3 nós) para alta disponibilidade

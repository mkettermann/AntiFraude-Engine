# ADR-004: Estratégia de Deployment — Docker Compose (Local) e Kubernetes (Produção)

**Status:** Aceito  
**Data:** 2026-04-23  
**Autores:** Time de Arquitetura

---

## Contexto

O AntiFraude Engine é composto por dois serviços (.NET) e duas dependências de infraestrutura (PostgreSQL, RabbitMQ). A estratégia de deployment precisa:

1. **Desenvolvimento local:** ser executada sem conta de cloud, em qualquer laptop do time
2. **Produção:** suportar alta disponibilidade, rolling updates, observabilidade e secrets gerenciados
3. **CI/CD:** ser fácil de integrar em pipelines automatizados

Opções consideradas:

- **Docker Compose apenas:** simples, mas não adequado para produção
- **Docker Swarm:** orquestração leve sobre Docker
- **Kubernetes:** orquestrador padrão da indústria, complexidade maior
- **PaaS (App Service, Cloud Run):** menor controle, mais gerenciado

---

## Decisão

**Docker Compose para desenvolvimento local + Kubernetes para produção.**

### Docker Compose (local)

- Arquivo em `infra/docker-compose.yml`
- 4 serviços: `postgres`, `rabbitmq`, `api`, `worker`
- Health checks entre serviços (`depends_on.condition: service_healthy`)
- Volumes nomeados para persistência de dados do Postgres e RabbitMQ
- Builds multi-stage com imagens `mcr.microsoft.com/dotnet/aspnet:9.0` (runtime mínimo)

### Kubernetes (produção — guidelines)

A solução foi projetada para ser Kubernetes-ready desde o início:

| Prática | Implementação |
|---|---|
| **Health checks** | `/health/ready` (readiness probe) e `/health` (liveness probe) |
| **Configuração via env vars** | Todas as strings de conexão e secrets via variáveis de ambiente |
| **Sem estado local** | API e Worker são stateless — podem ser escalados horizontalmente |
| **Graceful shutdown** | .NET `IHostApplicationLifetime` + `CancellationToken` em todos os consumers |
| **Imagens OCI** | Dockerfiles multi-stage produzem imagens mínimas (~100MB) |
| **Secrets** | Gerenciados via Kubernetes Secrets ou Vault (não hardcoded) |

**Recursos recomendados para produção:**

```yaml
# api-deployment.yaml (referência)
resources:
  requests:
    memory: "128Mi"
    cpu: "100m"
  limits:
    memory: "512Mi"
    cpu: "500m"
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
```

**PostgreSQL em Kubernetes:** recomenda-se o operador [CloudNativePG](https://cloudnative-pg.io/) para gerenciamento de cluster com failover automático.

**RabbitMQ em Kubernetes:** recomenda-se o [RabbitMQ Cluster Operator](https://www.rabbitmq.com/kubernetes/operator/operator-overview) para cluster de 3 nós.

## Consequências

### Positivas

- **Desenvolvimento local sem friction:** `docker compose up` levanta todo o ambiente em menos de 30 segundos
- **Portabilidade:** a mesma imagem Docker roda localmente e em qualquer cluster Kubernetes
- **Zero config de runtime:** todas as configurações via env vars — sem arquivos de config específicos por ambiente
- **Observabilidade integrada:** `/metrics` e `/health` prontos para scrape pelo Prometheus Operator e para probes do Kubernetes

### Negativas

- **Kubernetes não incluso:** os manifests Kubernetes não estão neste repositório; precisarão ser criados em um repositório separado de infra (GitOps)
- **Complexidade operacional:** Kubernetes requer expertise do time de plataforma para gerenciar upgrades, networking e storage
- **Docker Compose não é produção:** não deve ser usado em ambientes de staging ou produção

### Neutras

- Em ambientes de staging, o Docker Compose pode ser adaptado com imagens pré-buildadas (não builds locais)
- A variável `ASPNETCORE_ENVIRONMENT` controla o comportamento (Development vs Production); em produção, nunca deve ser `Development`

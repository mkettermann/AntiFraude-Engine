# ADR-003: Estratégia de Idempotência com Idempotency-Key e Tabela ProcessedMessages

**Status:** Aceito  
**Data:** 2026-04-23  
**Autores:** Time de Arquitetura

---

## Contexto

Em sistemas financeiros, retries de requisição são inevitáveis: timeouts de rede, falhas de cliente, ou políticas de retry de gateways de pagamento podem fazer com que a mesma transação seja enviada múltiplas vezes. Processar uma transação duas vezes pode resultar em decisões contraditórias ou cobranças duplicadas.

O desafio é garantir que um `POST /transactions` com os mesmos dados produza exatamente o mesmo resultado, independentemente de quantas vezes seja enviado.

Opções consideradas:

1. **Idempotência por `transactionId`:** usar o campo `transactionId` do body como chave
2. **Idempotência por `Idempotency-Key` no header:** chave explícita fornecida pelo cliente
3. **Idempotência a nível de consumer:** verificar duplicatas apenas no Worker (após desqueue)
4. **Cache distribuído (Redis):** armazenar respostas em cache com TTL

---

## Decisão

**Header `Idempotency-Key` + tabela `ProcessedMessages`** foi escolhido com o seguinte fluxo:

```
POST /transactions
  Header: Idempotency-Key: <uuid-gerado-pelo-cliente>

API:
  1. Calcula SHA-256(Idempotency-Key) → hash de 64 chars
  2. Busca hash em ProcessedMessages
  3a. NÃO encontrado: processa + salva ProcessedMessage(hash, responsePayload) → HTTP 202
  3b. ENCONTRADO:     retorna ResponsePayload salvo → HTTP 200 (sem reprocessamento)
```

**Por que SHA-256 e não o valor bruto?**

- Normaliza chaves de tamanho arbitrário para 64 caracteres fixos
- Evita armazenar dados potencialmente sensíveis do cliente no banco
- Constraint `UNIQUE` funciona eficientemente em coluna de tamanho fixo

**Garantia de atomicidade:** `Transaction` + `OutboxMessage` + `ProcessedMessage` são gravados na mesma transação de banco. Se qualquer parte falhar, nenhuma é persistida — garantindo consistência total.

---

## Consequências

### Positivas

- **Proteção contra duplicatas na borda:** verificação ocorre na API, antes de qualquer enfileiramento
- **Resposta determinística:** cliente sempre recebe a mesma resposta para a mesma chave, mesmo que a transação já tenha sido `REJECTED`
- **Atomicidade:** constraint `UNIQUE` no banco + transação atômica eliminam race conditions sem necessidade de locks distribuídos
- **Sem dependência de Redis:** não adiciona infraestrutura extra; a tabela `ProcessedMessages` é simples e eficiente com índice btree

### Negativas

- **Header obrigatório:** clientes que não implementam `Idempotency-Key` recebem `HTTP 422`; exige alinhamento com consumidores da API
- **Crescimento da tabela:** `ProcessedMessages` cresce indefinidamente; implementar job de limpeza para registros com mais de X dias (não incluído nesta versão)
- **TTL não gerenciado:** ao contrário do Redis, não há expiração automática; a resposta cacheada permanece para sempre

### Neutras

- A chave deve ser gerada pelo cliente (UUID v4 recomendado), não pela API
- Documentar claramente no contrato da API que `Idempotency-Key` deve ser única por tentativa de transação, não por cliente

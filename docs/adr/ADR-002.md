# ADR-002: Escolha do PostgreSQL como Banco de Dados Relacional

**Status:** Aceito  
**Data:** 2026-04-23  
**Autores:** Time de Arquitetura

---

## Contexto

O AntiFraude Engine precisa persistir:

- Transações com seus estados e decisões
- Trilhas de auditoria por transação
- Tabela de idempotência (`ProcessedMessages`)
- Outbox de mensagens pendentes de publicação

Os dados têm estrutura bem definida, exigem consistência ACID (especialmente para a constraint `UNIQUE` da idempotência) e precisam suportar queries relacionais (join de transação + auditoria).

Opções consideradas:

- **PostgreSQL** — RDBMS open-source, ACID, extensível
- **SQL Server** — RDBMS Microsoft, bom ecossistema .NET
- **MongoDB** — banco de documentos NoSQL
- **SQLite** — banco embutido, adequado apenas para desenvolvimento

---

## Decisão

**PostgreSQL 16** foi escolhido como banco de dados, com as seguintes práticas:

- Entity Framework Core com Npgsql como provider
- Migrations versionadas em `AntiFraude.Infrastructure/Data/Migrations/`
- **Migrations NUNCA são aplicadas automaticamente na startup em produção** (ver [Como Rodar](../../README.md))
- Índice `UNIQUE` em `ProcessedMessages.IdempotencyKeyHash` para garantir idempotência a nível de banco
- Volumes Docker nomeados (`postgres_data`) para persistência de dados entre restarts

---

## Consequências

### Positivas

- **ACID completo:** transações atômicas garantem que transação + outbox + idempotência são gravados juntos ou não são gravados
- **Constraint UNIQUE:** protege contra race conditions na idempotência sem necessidade de locks em aplicação
- **EF Core + Npgsql:** excelente suporte, migrations automáticas, LINQ queries tipadas
- **Open-source:** sem custos de licença; operado facilmente em Kubernetes via Helm chart ou operador CloudNativePG
- **JSON nativo:** suporte a `jsonb` para campos como `Metadata` sem overhead de serialização

### Negativas

- **Operação em produção:** requer expertise para tuning, replicação, backups e failover; em cloud, preferir RDS/Aurora ou CloudNativePG
- **Escalabilidade de escrita:** sharding horizontal é complexo; para volumes extremos, considerar CockroachDB ou Vitess
- **Migrations manuais:** exige processo de deploy disciplinado para garantir que migrations sejam aplicadas antes do código novo

### Neutras

- SQL Server seria igualmente viável no ecossistema .NET; PostgreSQL foi preferido por ser open-source e ter melhor suporte em cloud-native
- MongoDB foi descartado pois a consistência transacional entre `Transaction` e `ProcessedMessage` na mesma operação é crítica
